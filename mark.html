Numeração de Versões

A numeração de versões é fundamental para o gerenciamento do ciclo de vida de nossos projetos. O padrão que usamos para numeração de versões é o Semantic Versioning (SemVer) e adicionalmente a versão de compatibilidade com o Minecraft. O SemVer é composto por três partes principais: MAJOR, MINOR e PATCH e a versão de compatibilidade é acrescentada ao final. Aqui está como interpretar e atualizar cada parte:

  • MAJOR: Aumente quando fizer mudanças incompatíveis na API ou alterações significativas que quebram a compatibilidade com versões anteriores.
  • MINOR: Aumente quando adicionar funcionalidades novas, de forma retrocompatível.
  • PATCH: Aumente quando corrigir bugs de forma retrocompatível.
  • SYNCH: Defina a versão do Minecraft à qual a versão do projeto é destinada (descartando o primeiro valor: 1).

Nota: O Sufixo v deve ser adicionado antes da numeração da versão.

Exemplo de incremento de versão:

  • MAJOR: Deve ser aumentado de v1.0.0.0 para v2.0.0.0 após uma mudança incompatível.
  • MINOR: Deve ser aumentado de v1.0.0.0 para v1.1.0.0 após adicionar novas funcionalidades.
  • PATCH: Deve ser aumentado de v1.0.0.0 para v1.0.1.0 após corrigir bugs.
  • SYNCH: Deve ser definido de v1.0.0.0 para v1.0.0.20.5 após tornar a o projeto compatível com a versão 1.20.5 do Minecraft.

Definição de Fases de Desenvolvimento

Para garantir um processo de desenvolvimento estruturado, é importante definir as fases de desenvolvimento do projeto. As duas fases mais comuns são:

  • Alpha: Durante esta fase, o projeto é testado internamente ou com um grupo seleto de usuários. O foco está na estabilidade básica e nas funcionalidades essenciais do software. A numeração de versão deve incluir um sufixo -alpha (ex: 1.0.0-alpha).
  • Beta: Após a fase alpha, o projeto passa para a fase beta, onde é testado mais amplamente com uma base de usuários maior. O foco está na estabilidade geral, usabilidade e correção de bugs. A numeração de versão deve incluir um sufixo -beta (ex: 1.0.0-beta).

Numeração de Builds

As builds são versões específicas do projeto que são compiladas a partir do código-fonte em um determinado momento. Para numeração de builds, você pode seguir um esquema que inclui:

  • Número de versão: Use a numeração de versão SemVer (MAJOR.MINOR.PATCH.SYNCH) para identificar a versão base da build.
  • Identificador de build: Adicione um identificador de build para diferenciar as diferentes builds da mesma versão. Isso pode ser um número de build sequencial, um hash de commit ou outra identificação única.

Exemplo de numeração de build:

  • v1.0.0.0-alpha-b0
  • v1.0.0.0-alpha-b1
  • v1.0.0.0-beta-b0
  • v1.0.0.0-beta-b1

Conclusão

A adoção de práticas consistentes de numeração de versões, definição de fases de desenvolvimento e numeração de builds é essencial para o gerenciamento eficaz do ciclo de vida do software. Certifique-se de comunicar e aplicar esses padrões em todos os projetos da Kyubion Studios para garantir consistência e clareza ao longo do tempo.
